2026-04-22 [nba][grading] Fixed second archive INSERT crash in `upsert_grades` in `grading/grade_props.py`. After the batch-split fix in commit `fd9c67d`, the archive INSERT failed with "column name or number of supplied values does not match table definition" (SQL Server error 213). Root cause: `SELECT dg.*` expands to `daily_grades` columns in their physical storage order, which diverges from `daily_grades_archive` physical order because both tables accumulated `ALTER TABLE ADD` columns at different points in time. Ordinal positions are set at ADD time and cannot be changed. Fixed by replacing `SELECT dg.*` with an explicit column list (`_ARCHIVE_COLS` module constant) in both the INSERT target and the SELECT source, so the INSERT is immune to physical column order on either table. Commit `b452682`. See grading/ (no component README exists yet).

2026-04-22 [nba][grading] Fixed crash in `upsert_grades` in `grading/grade_props.py`. The archive/prune block after the MERGE had two CTE-based statements (WITH...INSERT and WITH...DELETE) inside a single `conn.execute()` call wrapped in `IF...BEGIN...END`. SQL Server via pyodbc treats each `execute()` as one batch; a `WITH` clause must be the first statement of its batch, so the second `WITH` caused a syntax error. Split into two separate `conn.execute()` calls. The `IF OBJECT_ID` guard was replaced with a scalar count check before the two calls so the logic is the same. Commit `fd9c67d`. See grading/ (no component README exists yet).
